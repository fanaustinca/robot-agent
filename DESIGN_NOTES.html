2. Object Detection: Gemini Verification  TODO

Current State

yolo_ik_agent/detect.py uses GroundingDINO (open-vocabulary zero-shot detector, despite "YOLO" naming throughout the codebase). It takes a text prompt and finds matching objects.

The text prompt comes from naive string parsing in yolo_ik_agent.py:

  • Verb detection: substring match against ["pick up", "pickup", "grab", "retrieve", "fetch", "collect"]
  • Noun extraction: everything after the keyword, with articles stripped
  • Example: "pick up the red lego brick" → target label "red lego brick"

Problems

Failure ModeWhat Happens
Wrong object has highest confidenceArm picks it up silently — no verification
Multiple similar objectsNo way to distinguish ("the red one, not the blue one")
Object not in framePicks whatever scores highest, or aborts with no retry
Ambiguous/spatial referencesGroundingDINO can't parse "the one on the left"

Solution: Gemini API as Post-Detection Verifier

User: "pick up the red lego brick" | v String parsing: extracts "red lego brick" | v GroundingDINO: runs on camera frame | returns N candidate detections with bounding boxes v Annotate frame with numbered bounding boxes | v Gemini API (gemini-3-flash-preview): "Which numbered box contains a red lego brick? Reply with the number, or 'none'." | v Use verified detection's center pixel for IK pipeline

What this solves:

  • Wrong-object picks — Gemini rejects incorrect detections
  • Multiple similar objects — Gemini understands color, size, spatial references
  • Nothing matches — Gemini returns "none", agent aborts gracefully
  • Adds ~1–2s latency (negligible vs arm movement time)

Rejected Alternatives

AlternativeWhy Rejected
REJECTED Gemini as pre-detection prompt refinerMarginal improvement over GroundingDINO's native text understanding. Extra API call not worth it.
REJECTED Local Gemma 4 modelGPU too constrained. FP16 won't fit alongside GroundingDINO. INT4 would fit but Gemini API is simpler and more capable. See hardware section.

3. Stereo 3D: Preset-Based Observe Positions  TODO

Current State

Two methods exist for computing 3D object position:

MethodCamerasHow It WorksLimitation
Table-planeTop onlyRay from pixel intersects Z=0 planeAssumes object is flat on table. Fails for cups, stacked objects.
Stereo triangulationTop + WristTwo rays from two cameras intersect in 3DWrist extrinsics from FK are inaccurate. Z is currently discarded.
Current workaround: In yolo_ik_agent.py line 362, triangulated Z is thrown away:
point[2] = GRIPPER_CLEARANCE  # Z from stereo is unreliable

Why FK-Based Wrist Extrinsics Are Unreliable

  • URDF link lengths and offsets must be perfectly accurate
  • Joint backlash and servo imprecision compound through the kinematic chain
  • Camera mount offset on the wrist must be precisely modeled
  • Small errors at each joint accumulate into large position errors at the camera

Solution: Pre-Allocated Observe Presets

Instead of computing wrist camera extrinsics dynamically from FK, define a small set of fixed arm positions. Each preset has its wrist camera extrinsics calibrated once using the existing checkerboard tool.

Top camera: object detected in zone "center-right" | v Lookup: zone "center-right" -> observe preset "right" | v Move arm to preset "right" (fixed joint angles) | v Wrist camera captures frame (extrinsics are pre-calibrated constants) | v Triangulate: top pixel + wrist pixel + known extrinsics | v Accurate 3D position (including height!) -> IK -> pickup

Preset Data Structure

OBSERVE_PRESETS = {
    "left":   {
        "joints": {"shoulder_pan": 30, "shoulder_lift": -42, ...},
        "cam_pos": [0.08, 0.19, 0.27],       # from calibration
        "cam_rot": [[...], [...], [...]],      # from calibration
    },
    "center": {
        "joints": {"shoulder_pan": 0, "shoulder_lift": -42, ...},
        "cam_pos": [-0.02, 0.22, 0.28],
        "cam_rot": [[...], [...], [...]],
    },
    "right":  {
        "joints": {"shoulder_pan": -30, "shoulder_lift": -42, ...},
        "cam_pos": [-0.12, 0.19, 0.27],
        "cam_rot": [[...], [...], [...]],
    },
}

Zone Mapping

Divide the top camera image into regions, each mapped to an observe preset:

Top camera view: +----------+----------+----------+ | | | | | zone_L | zone_C | zone_R | | | | | +----------+----------+----------+ | | | | | zone_FL | zone_FC | zone_FR | | | | | +----------+----------+----------+ zone_C -> observe_center (pan = 0) zone_L -> observe_left (pan = 30) zone_R -> observe_right (pan = -30) ...

Number of presets depends on wrist camera FOV. Start with 3 (left / center / right) and add more if coverage gaps appear.

Calibration Workflow (One-Time Per Preset)

  1. Move arm to the preset joint angles
  2. Run calibrate_extrinsic.py (already written, uses checkerboard visible to both cameras)
  3. Save resulting cam_pos + cam_rot in a JSON file keyed by preset name
  4. Repeat for each preset

Edge Cases

ScenarioHandling
Object at zone boundaryOverlap zones slightly; or try both presets and average
Wrist camera can't detect objectFall back to table-plane method (Z=0)
Top camera blocked by armWon't happen — top detection runs before arm moves to observe

Gemma 4 4B won't fit at FP16 (~5 GB) alongside GroundingDINO. INT4 quantized (~2.5 GB) would work but adds complexity. Decision: use Gemini API instead.
